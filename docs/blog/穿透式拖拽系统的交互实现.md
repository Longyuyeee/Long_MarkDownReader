# 深度交互设计：实现具备“穿透力”的文件拖拽系统

> **技术关键词**：拖拽交互 (Drag & Drop), 命中测试 (Hit Testing), document.elementsFromPoint, 影子 DOM, UX 优化

文件管理器的核心体验之一是“拖拽即移动”。但在自定义 UI 开发中，如何让系统准确识别出“拖到了哪个文件夹上”是一个经典的交互难题。

### 1. 现象描述
在实现文件拖拽功能时，我们通常会创建一个跟随鼠标移动的“影子 DOM（Drag Ghost）”来提示用户正在移动。但这会引发一个严重的副作用：无论鼠标移动到哪，鼠标尖下方的第一个元素永远是“影子”本身，导致下方的文件夹无法接收到悬停事件，无法高亮，也无法在松手后触发移动。

### 2. 原因分析
传统的鼠标命中测试（Hit Testing）遵循“首层优先”原则。
- **DOM 遮挡**：影子 DOM 虽然视觉上很薄，但在逻辑层面上它是一面完整的“墙”，阻断了与下方文件夹节点的物理接触。
- **pointer-events 局限性**：单纯设置 `pointer-events: none` 在某些快速移动或多层嵌套场景下，依然无法百分之百保证探测逻辑的穿透性。

### 3. 解决方案
为了打造无缝的穿透体验，我们采用了组合式方案：
1. **物理错位（Visual Offset）**：将影子 DOM 的位置设置在鼠标尖的右下方（如 `+10px`）。这样鼠标指针的点击中心点始终处于影子之外，排除了物理遮挡。
2. **多层探测算法（Multilayer Scan）**：
   使用浏览器提供的 `document.elementsFromPoint(x, y)` API。与只返回首个元素的 API 不同，它会返回该坐标下**所有**层级的元素数组。
   我们在代码中遍历该数组，跳过标识为“影子”的元素，直到找到带有 `data-drop-path` 的树节点。

### 4. 经验总结
交互开发的本质是模拟真实物理世界的反馈。当视觉效果（影子）干扰了物理反馈（目标探测）时，开发者需要通过**“坐标偏移”和“多层扫描”**等手段，为逻辑层开启一双能够“透视”的眼睛。
