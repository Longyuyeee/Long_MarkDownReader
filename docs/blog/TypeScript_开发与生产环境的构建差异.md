# TypeScript 构建策略：开发环境与生产环境的编译差异

> **技术关键词**：TypeScript, vue-tsc, 生产环境构建 (Build), 静态类型检查, 代码规范

在现代前端开发中，经常会出现“代码在开发模式正常，但在打包构建时报错”的情况。这背后的核心原因通常是生产环境开启了更严格的静态检查。

### 1. 现象描述
在 MistyEdit 项目的 v0.3.0 打包过程中，`npm run build` 频繁因为以下两类错误中断：
- **TS6133**: 定义了变量但从未被读取。
- **TS2554**: 函数调用时传递的参数个数与定义不匹配。

然而，这些代码在 `npm run dev`（热更新预览模式）下完全运行正常。

### 2. 原因分析
在生产构建流程中，Tauri 的默认配置会调用 `vue-tsc --noEmit`。这是一款专门针对 Vue 组件进行严格类型检查的工具。
- **开发环境**：为了提升编译速度，Vite 通常会跳过深层的类型校验，只关注代码的转换和运行。
- **生产环境**：为了确保生成的二进制文件具备最高的稳定性和最小的运行开销，编译器要求代码必须符合 TypeScript 的 100% 定义规范。任何多余的变量或错误的调用参数都会被视为潜在的逻辑漏洞。

### 3. 解决方案
针对此类错误，必须执行代码层面的“物理清理”：
1. **清理多余参数**：重构函数 definition，移除不再使用的参数。
2. **规范函数调用**：确保前端调用与 TS 接口声明保持一致。
3. **构建前置验证**：养成在提交代码前本地执行一次 `npm run build` 的习惯，而不是依赖热更新预览。

### 4. 经验总结
严谨的代码习惯不仅是为了美观，更是为了通过生产环境的“严格考核”。将**全量构建（Full Build）**纳入开发周期的一个环节点，能极大降低最终交付时的不确定性风险。
