# 跨应用标识符的数据自动化迁移方案

> **技术关键词**：Tauri 2.0, AppHandle, PathResolver, 文件系统, 自动化迁移

在应用进行品牌重塑或标识符（Identifier）变更时，会导致操作系统分配的默认存储路径发生位移。

### 1. 现象描述
当应用的 `identifier` 从 `com.mistyedit.mdhelper` 变更为 `com.longyuye.mdreader` 后，Tauri 提供的 `app_config_dir` 和 `app_cache_dir` 会指向全新的物理路径。这导致老用户在更新版本后，原本的 `config.json` 和历史记录（影子副本）会因路径不匹配而“丢失”。

### 2. 原因分析
操作系统的 `AppData/Roaming` 目录是通过标识符进行隔离的。简单地修改配置文件并不能触发物理目录的改名或搬迁。如果在应用启动时不进行干预，新版本将只能读取到空的默认配置。

### 3. 解决方案
在 Rust 侧实现一套启动时的探测与搬迁逻辑：
1. **路径解析**：利用 `app.path().app_config_dir()` 获取新路径，并通过字符串替换构造出旧路径。
2. **存在性检查**：判断旧路径是否存在且新路径是否尚未初始化。
3. **物理搬迁**：使用 `fs::rename` 进行原子化的目录移动。
4. **时机控制**：必须在 `setup` 闭包的最早期（如读取配置之前）执行，确保后续的 `get_config` 能够无感读取到迁移后的数据。

### 4. 总结
自动化迁移是保障用户体验“连续性”的关键。通过在后端屏蔽路径变更的物理差异，可以将重构带来的负面影响降至最低。
