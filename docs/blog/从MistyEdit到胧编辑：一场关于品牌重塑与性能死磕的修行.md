# 从 MistyEdit 到 胧编辑：一场关于品牌重塑与性能死磕的修行

### 背景
随着版本迭代，原本的 MistyEdit 正式升级为“胧编辑”。这次升级不仅是名称的变更，更是对“Markdown 编辑器”作为系统级工具稳定性的一次全方位大考。

### 那些年我们踩过的坑

#### 1. 控制台的“黑框快闪”
在实现系统检测功能时，由于 Rust 后端频繁调用 PowerShell，导致用户界面上时不时闪过一个黑色窗口。虽然只有几百毫秒，但对于追求“果味”视觉的胧编辑来说，这无异于一种亵渎。
**感悟**：在 Windows 桌面开发中，静默执行（Creation Flags）不是可选项，而是优雅的入场券。

#### 2. 设置界面的“响应式陷阱”
我们原本以为 Vue 3 的 `watch` 是自动化的救星，但在处理复杂的配置持久化时，它却成了死循环的推手。当“加载”与“保存”共用同一个响应式源时，必须有一道防线来区分“数据初始化”与“用户手动交互”。
**经验**：永远不要低估 `deep: true` 带来的副作用，给异步数据加载加个“锁”是架构师的基本操守。

#### 3. 消失的 MSI 安装包
当我们将产品名改为“胧编辑”后，WiX 工具链罢工了。原因竟然是它无法处理路径中的中文字符。
**策略**：面对老旧工具链的编码局限，显式指定语言环境（zh-CN）比盲目尝试路径转义更高效。

### 总结
从 v0.3.1 到 v0.3.4，我们解决的不只是 Bug，更是对 Windows 底层逻辑与前端状态机冲突的深刻理解。
胧编辑，不仅仅是好用的 Markdown 编辑器，更是极致稳定、无感运行的系统原生助手。
